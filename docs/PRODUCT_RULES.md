# AIR4 — Product Constitution

> Это не список правил. Это то от чего нельзя отступать.
> Читается перед каждым решением о разработке.

---

## Манифест

AIR4 — intelligent decision layer. Не трекер. Не дашборд. Не мотивационный бот.

Его единственная задача: помочь пользователю увидеть то что он сам не замечает — и сделать следующий шаг. Не информировать. Не анализировать ради анализа. Помочь решить.

Всё что не служит этой задаче — лишнее.

---

## Семь законов

**I. Один цикл.**
```
event → memory → observation → decision help
```
Каждая фича укрепляет этот цикл или не нужна вообще.

**II. Finance первым.**
AIR4 делает одну вещь отлично. Не пять вещей хорошо. Пока Finance не работает идеально — ничего нового не добавляется.

**III. Данные до вывода.**
Нет данных — нет вывода. Нет вывода — молчание. Молчание лучше банальности. Всегда.

**IV. Local first.**
Данные пользователя не покидают устройство без его явного разрешения. Это архитектурный принцип. Не обсуждается.

**V. Редко но точно.**
Одно точное наблюдение в неделю лучше пяти банальных в день. Частота — враг доверия.

**VI. Observation ≠ fact.**
Каждое наблюдение — гипотеза с доказательствами. Confidence system обязателен. Язык всегда соответствует уровню уверенности. Никаких уверенных выводов из одной точки данных.

**VII. Действие обязательно.**
Observation без следующего шага — аналитика ради аналитики. Каждый вывод AIR4 заканчивается вопросом или конкретным действием. Без исключений.

## Что не строим

### Не строим пока core loop не отточен
```
Health модуль         — после Phase 6.5
Онбординг             — после Phase 6.5
Mobile                — Phase 9
Интеграции            — Phase 10
Другие банки          — Phase 10
```

### Не строим никогда
```
Геймификация          — badges, streaks, points, leaderboards
Social features       — шеринг, сравнение с другими пользователями
Реклама               — любая форма
Продажа данных        — любая форма
Облачное хранение     — без явного согласия пользователя
Автодействия          — AIR4 не делает ничего от имени пользователя без подтверждения
Абстрактные скоры     — "Life Score: 73/100" бессмысленно и создаёт тревожность
```

## Красные флаги

Видишь это — останавливаешься.

### В продукте
```
🚩 AIR4 говорит что-то каждый день
🚩 Observation без вопроса или действия в конце
🚩 "Ты мало спишь" — вывод без данных и паттерна
🚩 AIR4 хвалит пользователя просто так
🚩 Уверенный вывод из одного data point
🚩 Badge или streak за активность
🚩 Push уведомление без реального сигнала
🚩 "Я понимаю как это тяжело" — AIR4 не психолог
```

### В архитектуре
```
🚩 Весь контекст пользователя в каждый запрос
🚩 Observation Engine только на LLM — нет rule layer
🚩 Нет confidence score у observation
🚩 Нет Gate Check перед генерацией observation
🚩 Один системный промпт для всех агентов
🚩 Нет memory lifecycle — данные копятся бесконечно
```

### В разработке
```
🚩 Новая фича до полировки существующего
🚩 Интеграция до валидации ручного режима
🚩 "Давай добавим ещё одну сферу" — без данных о качестве текущих
🚩 Дизайн без реальных данных — макеты с placeholder content
```

---

## Критерий готовности

Один. Единственный.

```
"Чёрт… он прав."
```

Пользователь смотрит на вывод AIR4 и понимает что сам бы это не увидел.

Не DAU. Не retention. Не количество observations. Не красивый дизайн.

Если этого момента нет — всё остальное не имеет значения.

---

## Приоритизация — три вопроса

Перед каждым решением о разработке:

**1. Это укрепляет core loop?**
Нет → не берём.

**2. Это улучшает качество существующего или добавляет новое?**
Улучшение → приоритет. Новое → только если текущее уже работает хорошо.

**3. После этого пользователь скорее скажет "чёрт… он прав"?**
Нет → не берём.
